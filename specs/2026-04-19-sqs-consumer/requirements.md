# Requirements — Phase 8: SQS Consumer

_Branch: `phase-8-sqs-consumer` | Date: 2026-04-19_

---

## Scope

### Included

Implement a long-poll SQS consumer in `services/central/consumer.py` that:

1. **Dequeues messages** from an SQS queue using boto3 long-polling
   (`WaitTimeSeconds=20`)
2. **Invokes `SENTINEL_GRAPH`** for each message, passing the query extracted
   from the message body
3. **Writes an audit log record** to `services/api/task_store` (the in-memory
   store from Phase 7) on every graph completion — keyed by the task ID
   embedded in the message
4. **Handles at-least-once delivery** via an idempotency check: if a task ID
   already exists in the store, skip re-processing and delete the message
5. **Deletes the message** from the queue after successful processing (or after
   idempotency skip) to prevent redelivery

For this phase, the SQS client is injected as a dependency so tests can pass a
fully stubbed boto3 client without LocalStack or real AWS credentials.

### Not included

- No real SQS queue or LocalStack — the SQS client is stubbed via
  `unittest.mock` in tests
- No real RDS — audit log is written to the Phase 7 in-memory `TASK_STORE`;
  Phase 10 provisions RDS
- No FastAPI endpoint changes — Phase 7's `GET /result/{task_id}` already reads
  from `TASK_STORE`, so results written by the consumer are immediately readable
- No process management / supervisor loop — the consumer exposes a
  `consume_one()` function (processes a single poll cycle) and a
  `run_consumer()` loop; tests call `consume_one()` directly
- No Docker / ECS task wiring — Phase 10

## Message contract

Messages placed on the SQS queue by the API (Phase 7 stubbed this; Phase 8
defines the format):

```json
{
  "task_id": "<uuid4>",
  "query": "where is vessel IMO-001"
}
```

- `task_id`: the UUID generated by `POST /query` in Phase 7
- `query`: the raw query string from the original request

The consumer parses this JSON body, extracts `task_id` and `query`, invokes
the graph, and saves the result under `task_id`.

---

## Consumer behaviour

### `consume_one(sqs_client, queue_url: str) -> bool`

Performs one long-poll cycle:

1. Call `sqs_client.receive_message(QueueUrl=queue_url, MaxNumberOfMessages=1, WaitTimeSeconds=20)`
2. If no messages: return `False`
3. Extract `task_id` and `query` from the message body JSON
4. **Idempotency check**: if `task_id` already in `TASK_STORE`, delete message and return `True`
5. Invoke `SENTINEL_GRAPH` with `make_state(query)`
6. Save `TaskResponse` to `TASK_STORE` under `task_id`
7. Delete the message (`sqs_client.delete_message`)
8. Return `True`

### `run_consumer(sqs_client, queue_url: str) -> None`

Infinite loop calling `consume_one()` in a tight loop. Catches and logs
exceptions per iteration so one bad message does not crash the process.

---

## Error handling

| Scenario | Behaviour |
|----------|-----------|
| Malformed message body (not valid JSON) | Log error, delete message (poison pill), continue |
| Missing `task_id` or `query` in body | Log error, delete message, continue |
| Graph invocation raises | Log error, **do not delete** message (allow redelivery up to DLQ) |
| boto3 raises on receive | Log error, sleep 5 s, continue loop |

## Data flow

```
POST /query (Phase 7, currently synchronous)
    │
    ▼  [Phase 8 changes POST /query to enqueue instead of invoke]
SQS queue
    │
    ▼
consume_one()
    ├─ invokes SENTINEL_GRAPH
    ├─ saves TaskResponse to TASK_STORE[task_id]
    └─ deletes SQS message

GET /result/{task_id} (Phase 7, unchanged)
    └─ reads from TASK_STORE[task_id]
```

**Important**: Phase 7's `POST /query` currently invokes the graph
synchronously. Phase 8 updates it to enqueue the message instead, making the
full pipeline asynchronous. `GET /result/{task_id}` is unchanged — it reads
from `TASK_STORE` which is now written by the consumer.

## Decisions

**Dependency-injected SQS client**
Passing the boto3 client as a parameter (rather than creating it inside the
consumer) makes the consumer fully testable without LocalStack. Tests pass a
`MagicMock` that returns canned `receive_message` payloads.

**`consume_one()` + `run_consumer()` split**
Tests call `consume_one()` directly. The infinite `run_consumer()` loop is
wired up at process entry and is only tested via integration smoke tests.
This avoids needing to interrupt an infinite loop in unit tests.

**In-memory audit log (no RDS)**
Phase 10 provisions RDS. The consumer writes to `TASK_STORE` (Phase 7's
in-memory dict). Since `GET /result/{task_id}` already reads from it, the
API layer works end-to-end without any schema changes.

**Poison pill deletion**
Malformed messages are deleted immediately so they don't clog the queue. A
real DLQ would catch them in production; for this phase the behaviour is
explicit.

**`POST /query` updated to enqueue**
Phase 7 stubbed the SQS step. Phase 8 completes it: `POST /query` puts a
message on the queue and returns `{"task_id": ..., "status": "pending"}`.
The consumer processes it asynchronously. `GET /result/{task_id}` returns
`status: "pending"` until the consumer writes the result.

## Changes to `services/api/main.py`

`POST /query` updated:

```python
@app.post("/query", status_code=202, response_model=TaskResponse)
def post_query(request: QueryRequest) -> TaskResponse:
    task_id = str(uuid.uuid4())
    # Enqueue message instead of invoking graph directly
    sqs_client.send_message(
        QueueUrl=settings.queue_url,
        MessageBody=json.dumps({"task_id": task_id, "query": request.query}),
    )
    task = TaskResponse(
        task_id=task_id,
        status="pending",
        intent=None,
        agent_result=None,
        error=None,
    )
    save_task(task_id, task)
    return task
```

The SQS client and queue URL are injected via a `Settings` object (Pydantic
`BaseSettings` reading from environment variables). In tests, the client is
mocked.
